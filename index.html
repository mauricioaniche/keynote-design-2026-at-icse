We love software design

and that is why we tend to think this is the most important part of software.

The uncomfortable truth:

at scale, the real killers are architecture, infrastructure, and data. Not code.

Bad code is unpleasant, but survivable.

Local messes can be cleaned slowly and independently.

In an agentic world, code is even less of a problem.

Local messes get cheaper to read, patch, and reshape.

Every code and architectural decision expires.

The real question is not the 'what', but the 'when'.

Let’s get concrete:

a more practical design philosophy

Optimize for reversibility, not timelessness.

Get comfortable with half-life designs.

Make failure visible.

If something will hurt, let it hurt in a way you can detect, localize, and roll back.

Treat data and infrastructure as first-class.

They outlive code fashion. They are where migration cost concentrates.

Take the simplest path and do not overengineer.

The future will prove you wrong 100% of the time.

Go for code abstractions only when you have evidence you need them.

Simpler code is always better.

Be ready to refactor.

not too soon but not too late.

A summary

  • Every design decision is a trade-off under uncertainty.
  • At scale, architecture and infrastructure hurt.
  • Refactoring and migration are the core large-scale design skills.
  • At code level, concurrency, error handling, workflow enforcement, service-to-service communication is more important than code beauty.
  • Our abstractions should match the real shape of the work, simplicity is key.
  • The goal is not perfect design, but refactor at scale.
